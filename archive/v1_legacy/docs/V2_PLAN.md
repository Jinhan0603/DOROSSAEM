# Simplified Operating Plan

## 1. 목표

- 강사 DB를 `최소 입력 + 활동 로그 + 역할/등급 결과` 구조로 단순화한다.
- 매니저는 강사의 기본 정보만 넣고, 나머지는 로그가 쌓이면서 자동 갱신되게 만든다.
- 강사 DB의 목적은 "누가 어떤 역할에 바로 투입 가능한가"를 빠르게 판단하는 것이다.

## 2. 핵심 구조

### 2.1 Instructor_Master

매니저가 보고 판단하는 메인 시트.

필수 컬럼:

- `고유ID (instructor_id)`
- `이름`
- `최초 기수`
- `활동 상태 (Status)`
- `등급 (Tier)`
- `보조강사 배정`
- `주강사 배정`
- `DLS 주강사 배정`
- `운영인력 배정`
- `강의 참여 점수`
- `강의 피드백 점수`
- `기여 로그 점수`
- `패널티 점수`
- `최종 점수`
- `학교`
- `학과`
- `전공/프로젝트 경험`
- `거주지`
- `이동 가능 범위`
- `최근 업데이트`

### 2.2 Instructor_Sensitive

민감정보 분리 시트.

- `전화번호`
- `주민번호`
- `이메일`
- `은행`
- `계좌번호`
- `주소`

### 2.3 Activity_Log

모든 활동을 쌓는 append-only 로그 시트.

필수 컬럼:

- `logged_at`
- `instructor_id`
- `source`
- `category`
- `activity_type`
- `role`
- `points`
- `penalty_points`
- `request_id`
- `note`

## 3. 활동 로그 원칙

강사의 활동은 전부 `Activity_Log`로 받는다.

예시:

- OT 참여
- 콘텐츠 교육 수강
- 수업 참여
- 교안 제작
- 교구재 제작
- OT 강의
- 교육 영상 제작
- 포장 알바
- 추천
- 월간 우수 선정
- 패널티

입력 채널은 나중에 늘려도 된다.

- Google Form
- Discord bot
- Notion DB sync
- 매니저 수기 입력

중요한 것은 채널이 아니라, 최종적으로 `Activity_Log` 한 곳에 적재되는 구조다.

## 4. 역할 기준

강사 등급보다 먼저 보는 것은 역할 적합성이다.

- `보조강사 배정`
- `주강사 배정`
- `DLS 주강사 배정`
- `운영인력 배정`

역할 컬럼은 "배치 가능 여부"만 명확하게 보여주면 된다.

## 5. 등급 기준

등급은 성장 단계와 신뢰도를 보여주는 운영용 라벨이다.

- `General`
- `Advanced`
- `Master`
- `Penalty`

운영 해석:

- `General`: 기본 배치 가능, 성장 중
- `Advanced`: 주강사 중심으로 안정적 배치 가능
- `Master`: 운영인력 포함 핵심 인재
- `Penalty`: 배치 제한 또는 재검토 대상

`Master`는 월간 업데이트 시점에 상위권 중심으로 제한 적용해도 된다.

## 6. 점수 구조

점수는 하나만 본다.

`최종 점수 = 강의 참여 점수 + 강의 피드백 점수 + 기여 로그 점수 - 패널티 점수`

세부 계산보다 중요한 것은:

- 어떤 활동이 점수에 반영됐는지 설명 가능할 것
- 패널티가 누락되지 않을 것
- 역할 배정 판단에 바로 쓸 수 있을 것

## 7. 지금 버리는 것

이번 단순화 기준에서 바로 버린다.

- `N/G/A/M`
- Assignment/Promotion 이중 점수
- Qualification, Availability 별도 운영
- Evidence Pack 중심 승급 구조
- 과도한 보조 테이블

## 8. 현재 우선순위

1. 기본 강사 DB 단순화
2. Activity_Log 적재 구조 고정
3. 폼 4개 + Activity_Log + Instructor_Master 연결
4. 역할/등급 자동 갱신
